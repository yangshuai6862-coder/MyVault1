亚马逊商品安全问题绩效主要逻辑如下

### 1.风险是否真实存在

→ 有没有“证据级事故”（照片 / 视频等证据）

### 2.产品是否真实存在问题

→ 自查Root Cause，可以合理怀疑用户使用不当导致安全问题发生，但不能完全甩锅用户。

### 3.是否“已经控制风险”

→ 针对可能导致问题产生的风险，是否有进行改善或者改善计划, 如Listing / 产品 / 包装 / 流程是否改变

---

# 处理流程

### 1. 立即风险评估

内部自查过往客诉、邮件等：

客诉记录（Review / Message / Return reason）

是否有关键词：

- explosion / fire / burn / overheat
    

输出：是否存在“真实事故证据”

**同步整理产品认证证书、报告等资料，按检测项目命名并清晰的分类整理好，以便及时提交给对应的case，证明我们自身的安全性。（****针对3C带电的产品，一定需要定期做相关认证（亚马逊要求三年内的报告），以随时应对投诉和产品安全问题****）--可以找产品部们同时协助处理此流程。****按亚马逊流程，前期投诉仅需提供相关检测报告和证明文件即可通过申诉。如后续仍有新投诉增加，则需要按亚马逊邮件要求提交POA处理。**

### 2. 启动“Listing风控处理”（如Listing未被冻结停售）

**必须立刻做（不等POA）：**

商品页面更新充电警示、充电指引等内容。

### 3. 尝试联系投诉买家（证据关键）

目的：

- 获取证据（照片/视频）
    
- **如无回复，或无法联系到买家，即证明“无法验证事故”**
    

### 4.Root Cause 分析（核心）

**卖家角度的 Root Cause 分析**

设计（Design）

生产（Manufacturing）

包装（Packaging）

运输（Logistics）

使用（Usage）

逻辑是先理清产品本身上各个环节上可能导致问题的点，进一步分析这些点我们是怎么处理，会不会有风险，说清楚产品在我们的这些措施下，是不存在安全风险即可,用事实依据佐证，排除产品自身原因，但是不能直接言语表达我们没有问题，POA应避免主观判断、定性。

**买家角度的 Root Cause 分析**

是否存在使用不当或没有按正确方式充电的角度去分析。尽量使用"可能"、“也许”这种推测、怀疑的语气去阐述，不能完全甩锅给买家。

### 5. Plan of Action

从ROOT CAUSE分析的结果来整理。

在第四点root cause分析中整理的资料也佐证了产品本身没有问题，所以这里只需要针对买家角度作出相应的改善和解决措施以减少或防止此类问题再次发生。卖家角度的root cause分析详细全面的证明了自己的产品没问题，买家角度要体现出我们想解决问题，也在尝试解决问题。

  

### 6. 其他注意事项

①申诉时提及的资料、其他证明文件等内容少的可以直接放在原文中，内容偏多则整理为附件，在POA原文中提及附件1、附件2的形式作为补充，避免出现在POA原文，影响整体阅读性。

②Root cause部分内容，一定要注意，原因都一律以怀疑、可能这种表达，传达出是我们的推测，不能太主观。

（Amazon所有申诉POA都应注意这一点以提高POA通过率）

③Listing停售等重大绩效问题，除绩效case渠道提交POA外，同步POA给亚马逊付费经理，走内部渠道提交POA，加快处理进度。

④常规情况结合此文档，按亚马逊要求提交相应POA即可。 特殊情况下，怀疑是竞对恶意投诉导致的绩效问题，需要另外补充一份单独的文档，说明出货记录、出货数量，以及过往销售情况等其他佐证过往未出现此类问题，列出近期为解决被竞对其他恶意攻击的行为而开启的CASE ID，引起审核人员合理猜测，敦促其加速处理此案件。

  

# POA 模版

**1.第一部分** 查看亚马逊官方邮件通知或绩效界面通知，按照要求填写涉事产品ASIN、ECN、UPC，Seller ID等

![[Attachments/POA模板_截图1.png]]

2.**第二部分**

**Executive Summary** 针对此次问题的调查和分析做简要总结。先“降风险感知”，再“表达负责态度”，最后“引出行动”。

3.**第三部分**

逐一回答亚马逊官方邮件的问题，行文逻辑清晰。一般前几个问题都是产品相关的问题，最后两个问题分别是root cause分析和Plan of Action。针对潜在问题，分点列出清晰的应对措施。并列出整改内容相应的附件，以PDF格式提交。
